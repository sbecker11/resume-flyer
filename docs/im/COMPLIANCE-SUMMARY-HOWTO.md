# How the IM Compliance Summary Was Generated (and How to Improve It)

## How the summary was generated

The **56% IM compliance (14/25 components)** line and the lists of ‚ÄúFully Compliant‚Äù vs ‚ÄúRemaining‚Äù components in [README.md](README.md) came from the **component scanner**, not from manual counting.

1. **Source of truth**  
   The scanner lives in **`archive/componentScanner.mjs`**. It:
   - Discovers component files (`.vue`, and `.mjs`/`.js`/`.ts` that match Controller/Manager/Service patterns).
   - For each file, analyzes content for IM patterns (e.g. `initialize(dependencies)`, `setupDom()`, template ref usage, `getElementById`) and records violations.
   - Builds:
     - `foundComponents` ‚Äì all detected components
     - `violatingComponents` ‚Äì components with one or more violations
   - Computes **compliance rate** as:  
     `(foundComponents.length - violatingComponents.length) / foundComponents.length * 100`  
     So ‚Äú14/25‚Äù means 14 compliant, 25 total.

2. **How the docs got the numbers**  
   At some point (e.g. when the scanner lived under `modules/core/` or when `archive/` had a runnable path), someone:
   - Ran the scanner (e.g. via a script or the server).
   - Took the printed **compliance rate** and **component lists** (compliant vs violating).
   - Pasted or adapted that output into **`docs/im/README.md`** (the ‚ÄúLatest Achievement‚Äù and component lists).  
   So the summary is **scanner output that was manually copied into the doc** and then left static.

3. **What the scanner can output**  
   The same file contains:
   - **`generateReport()`** ‚Äì builds a markdown report with total/compliant/violating counts, compliance rate, and per-component violation details.
   - **Console logging** in **`_analyzeResults()`** ‚Äì e.g.  
     `Compliance rate: 56.0% (14/25 components compliant)`.

So: the summary was generated by **running the component scanner and then copying its report/console output into the documentation**.

---

## Why the summary is stale and hard to regenerate

- The **live entry point** is **`scanProject()`**, which calls `scanFiles()`, `ReportGenerator.generateComplianceReport()`, etc. Those depend on **`./scanning/*`** modules (e.g. `ScannerConfig`, `CodeParser`, `DOMViolationDetector`), which **do not exist** under `archive/` (the `archive/scanning/` directory is missing). So **`scanProject()` cannot run** as-is.
- The **logic that actually computes the rate and lists** (e.g. `_scanFile()`, `_analyzeFileContent()`, `_analyzeResults()`, `generateReport()`) is still in **`archive/componentScanner.mjs`**, but there is **no working script or server path** that:
  - Uses that logic,
  - Runs a full scan,
  - And either prints the report or writes `this.scanResults` for the server.
- The **server** (`server.mjs`) has routes **`GET /api/check-dependencies`** and **`GET /violations`** that expect a `scanResults` (and optionally `report`) object, but the code that **runs the scan and sets those variables is missing** (only comments like ‚ÄúScan project for dependency violations‚Äù remain). So those endpoints cannot currently return real compliance data.

So the summary is **stale** because:
1. It was a one-time copy of scanner output.
2. There is no runnable path to produce that output anymore.
3. The server never runs the scanner to fill in the summary.

## How to improve it

### Option A: Make the scanner runnable again (recommended)

1. **Restore a single-file runnable path**  
   - Either move or copy the **working, self-contained scanning logic** from `archive/componentScanner.mjs` (the parts that use `_scanFile`, `_analyzeFileContent`, `_analyzeResults`, and `generateReport()`) into a script or a **single scanner module** that:
     - Does **not** depend on `./scanning/*`.
     - Discovers files (e.g. same rules as now: skip `node_modules`, `.git`, etc.).
     - For each file, runs the same analysis and builds `foundComponents` / `violatingComponents`.
     - Calls the same summary/report generation (compliance rate + lists).
   - Provide a **Node script** (e.g. `scripts/run-compliance-scan.mjs`) that:
     - Imports that scanner,
     - Runs the scan (e.g. `scanProject('.')` or equivalent),
     - Prints the markdown report to stdout (or writes to a file).

2. **Regenerate the summary**  
   - Run the script (e.g. `node scripts/run-compliance-scan.mjs`).
   - Copy the ‚ÄúSummary‚Äù and ‚ÄúViolating Components‚Äù (and optionally ‚ÄúCompliant‚Äù list) into **`docs/im/README.md`** in the ‚ÄúLatest Achievement‚Äù section, or‚Ä¶

3. **Optionally auto-update the doc**  
   - Have the script write a **small markdown fragment** (e.g. `docs/im/compliance-summary-fragment.md`) with:
     - Compliance rate and ‚ÄúX/Y components‚Äù
     - List of compliant components
     - List of violating components (and optionally one line per violation).
   - Then either:
     - Manually replace the summary block in `README.md` with the contents of that fragment, or
     - Add a note in `README.md`: ‚ÄúCompliance status is generated by `scripts/run-compliance-scan.mjs`; see `docs/im/compliance-summary-fragment.md`.‚Äù

### Option B: Wire the server so ‚ÄúCompliance Status‚Äù is live

1. **Implement the scan in the server**  
   - In `server.mjs`, for **`/api/check-dependencies`** and **`/violations`**:
     - Import (or require) a **working** scanner implementation that does not depend on missing `./scanning/*` modules.
     - Call the equivalent of `scanProject(projectRoot)` (or the same logic that builds `foundComponents` / `violatingComponents`).
     - Set **`scanResults`** (and **`report`** if you have a `generateReport()`-style function) from that result.
   - Keep the rest of the route logic as-is (return JSON for `/api/check-dependencies`, return HTML for `/violations`).

2. **Document how to get compliance**  
   - In **`README.md`** or **`docs/im/README.md`**, add a short ‚ÄúCompliance Status‚Äù section, for example:
     - ‚ÄúRun the dev server and open **`/api/check-dependencies`** for JSON or **`/violations`** for an HTML report.‚Äù
   - That way the ‚Äúsummary‚Äù is no longer a static snapshot; it‚Äôs whatever the scanner currently reports.

### Option C: Keep the doc as a snapshot, but document the process

- In **`docs/im/README.md`**, add a sentence where the 56% line lives, for example:  
  **‚ÄúThis snapshot was generated by the component scanner (see `docs/im/COMPLIANCE-SUMMARY-HOWTO.md`). To refresh it, run the scanner and update this section.‚Äù**
- In **`docs/im/COMPLIANCE-SUMMARY-HOWTO.md`** (this file), keep the ‚ÄúHow to improve it‚Äù steps above so that when the scanner is runnable again, anyone can regenerate or automate the summary.

## Quick reference

| Goal | Action |
|------|--------|
| **Understand where 56% (14/25) came from** | Component scanner in `archive/componentScanner.mjs`; numbers were copied from its output into the doc. |
| **Get current compliance** | Make the scanner runnable (Option A) and/or wire it in the server (Option B), then run the script or hit the HTTP endpoints. |
| **Keep the doc accurate** | Regenerate the summary after changes (Option A or B) and update `docs/im/README.md` (or the fragment file) as in Option A. |
